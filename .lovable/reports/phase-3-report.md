## Two-Dimension Reporting Model

Every metric distinguishes:

- **Dimension 1 — Households** (tracking numbers, submitted units, approval status).
- **Dimension 2 — Beneficiaries** (each secondary / university / college / TVET student).

Dashboard tiles include Households, Beneficiaries, Secondary, Higher Ed, Avg per Household, Disabled, Approved, Pending / Rejected, plus a full budget row (Requested, Recommended, Allocated, Remaining).

## Validation Rules Added (Phase 3)

Extended `src/lib/validation/consistency.ts` with three additive rules (still soft, never blocking):

| Code | Trigger |
| --- | --- |
| `day_school_high_fees` | Day-school declared but secondary fee balance > 30k. |
| `ncpwd_without_declaration` | NCPWD number provided but questionnaire says no disability. |
| `students_exceed_declared` | Declared child count < number of student records. |

Phase 1 rules preserved untouched: `orphan_vs_parents_alive`, `unemployed_with_income`, `boarding_no_fees`, `disability_incomplete`, `helb_without_higher_ed`.

## Export Improvements

- Household grouping preserved in both PDF and Excel — never flattened.
- Every export embeds a filter/audit block (generated at, generated by, filters used, version).
- Excel exposes both dimensions on separate sheets so pivots & Power-BI reconstruct the hierarchy from `Tracking`.

## Auditability

Every generated report ships with `GeneratedReportMeta { generatedAt, generatedBy, filters, format, version }`. PDF prints it in the header; Excel writes it to the Filters sheet.

## Performance

- Metrics, filtering and grouping are pure and memoised inside the panel via `useMemo`.
- No new RPC calls — reuses the existing `useHouseholds` hook and its 20s silent polling.
- Exports run entirely client-side; no server round trip.

## Outstanding Risks

1. `recommended_amount` per beneficiary is currently a client-side heuristic (20k secondary / 35k higher-ed). Once the AI allocation engine writes a recommended figure to the DB, swap to that value in `metricsEngine.recommendedFor`.
2. `academic_year` grouping uses the current year — extend to read from `bursary_adverts` once academic-cycle metadata is exposed.
3. `identifier` field on `HouseholdStudent` is not part of the RPC yet — duplicate detection currently only fires when the field is provided by upstream code. Follow-up: extend `get_parent_applications_for_commissioner` to return it.
